How Git Works, Visually

The mental model behind commits, branches, and merges

Pranjal Rawat

The problem, and the fix

Every result you trust rests on one question: exactly which version of the code and data produced it?

Git is the machine that gives that question an answer.

This deck shows the mental model, not the commands. By the end you should be able to picture what git is doing when someone says commit, branch, merge, or rebase.

Life without version control

This is the folder-of-dated-spreadsheets problem every data scientist has felt. Version control exists to answer all four of these questions at once.

What version control gives you

For a data scientist, these three add up to one word: reproducibility.

History, recovery, collaboration. The payoff for our work specifically is reproducibility: a result is only trustworthy if you can say exactly which version produced it.

A commit is a snapshot

The unit of git. One commit records three things.

Snapshot plus message plus parent pointer. The parent pointer is what turns a pile of commits into a history.

Commits form a chain

Each commit points back to the one before it. That is the whole history.

Arrows point from a commit back to its parent. The newest commit is where “latest” sits. Add a commit and the chain just grows to the right.

The hash is a fingerprint

Each commit’s ID is computed from its contents. Change the past and every ID downstream changes with it.

That is how git guarantees the history has not been quietly tampered with.

The hash is a fingerprint of everything that came before. This tamper-evidence is a free side effect of the parent-pointer design.

How to read the diagrams that follow

Many of the diagrams from here on are a photograph of a real git repository. A command is run, and git’s actual state is read back and drawn.

A commit is a circle. An arrow runs to its parent. A colored box is a branch (just a pointer). HEAD marks where you stand. An amber ring is the commit the last command just made.

Use the arrow keys. Each step is one command, and the graph grows as you go.

A change moves through three places

You edit, you choose what to record, then you record it.

Working directory, staging area, history. The staging step is the part people forget: it is what lets you commit a tidy logical change out of a half-finished mess.

Distributed: every copy holds it all

Clone a project and you copy the entire history to your machine, not just the latest files.

This is the heart of git’s design. There is no single authoritative server the way older centralized tools had. Every clone is complete.

git is not GitHub

A distinction worth keeping straight in any conversation. git is the tool; GitHub is a host built on it. Torvalds wrote git in 2005 to manage the Linux kernel after BitKeeper’s free terms were withdrawn.

Branches, and how they rejoin

A branch is a separate line of work: a movable pointer that lets you build without disturbing the main line.

A branch is just a movable label

The whole trick: a branch is a name pointing at a commit. Commit on it and the pointer advances. No files are copied.

The fork: work in isolation

Almost every change a modern team ships starts life on a branch like this. The main line stays stable while the work happens off to the side.

Merge: tie the lines together

A merge commit has two parents, one from each branch. Merging is usually automatic; git stitches together changes to different files or different parts of a file on its own.

Rebase: replay onto a straight line

Rebase rewrites your branch to look as if it were built on the latest main. Cleaner to read, but it changes commit IDs, so the rule is: rebase only your own un-shared work.

A merge conflict is not an error

Conflicts happen when two branches change the same lines incompatibly. They are more frequent and more painful the longer a branch lives apart, which is the practical argument for keeping branches short.

Working as a team

The mechanics of branching are settled. What teams argue about is the workflow: when to branch, how long a branch lives, when to merge.

Three workflows on one spectrum

Git-flow (Driessen, 2010) suits software shipped as separately supported versions. Trunk-based is what high-performing teams tend to use. GitHub flow sits in between and is the everyday default. In 2020 Driessen added a note that panaceas do not exist.

The pull request

Where a branch is reviewed, tested, and let into main.

This is the cultural payoff of cheap branching. The automated checks in the third box are continuous integration, covered in the sibling card.

The one tradeoff that runs backwards

Small, frequent merges beat one big one. This is the logic behind trunk-based development and feature flags.

The counterintuitive bit: the safe-feeling long-lived branch is the dangerous one. The useful question in a review is simply, how long has this branch been alive?

The workflow is not the mechanics. It is the set of rules a team agrees on: when to branch, how long a branch lives, when to merge. Four common shapes, each on a real repository.

The words you will actually hear

THE CORE
commit a recorded snapshot, with a message and a parent
hash the fingerprint that identifies a commit
staging area the changes marked for the next commit
clone a full local copy, history and all
remote another copy you sync with
push / pull send commits up / bring commits down
BRANCHING
branch a movable pointer, a line of work
main the line treated as the source of truth
merge join two lines, recording a merge commit
rebase replay commits for a linear history
merge conflict same lines changed two ways, a human decides
pull request a proposal to merge, where review happens

This is the real payoff of the whole deck. Knowing these twelve words is what lets you follow, and join, an engineering conversation about git.

The whole idea, in one line

A chain of snapshots,

each pointing at its parent.

Everything else, branches, merges, rebases, remotes, is bookkeeping on top of that.

If someone remembers one thing, it should be this. The commit graph is the object; all the commands are just ways of moving pointers around it.
